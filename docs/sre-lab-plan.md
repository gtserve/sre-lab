# SRE Lab — Project Plan

A self-contained, fully local Site Reliability Engineering showcase project,
designed to deepen SRE expertise and serve as a portfolio piece for senior
DevOps / SRE interviews.

---

## 1. Project Overview

**SRE Lab** is a GitOps-managed Kubernetes laboratory that practices the
four pillars of Site Reliability Engineering — Service Level Objectives,
observability, chaos engineering, and incident response — on top of a
real-world polyglot microservices application. The project is extended
with progressive delivery, an automation hub for toil reduction, policy
enforcement, cost visibility, and capacity planning.

Everything runs on a laptop using `kind` (Kubernetes-in-Docker). No cloud
account, no monthly bill, no vendor lock-in. The entire stack is open
source and reproducible from a single repository.

The project is presented as a **public GitHub monorepo** with extensive
written documentation (architecture diagrams, SLO specifications, runbooks,
postmortems, chaos experiment reports, and a capacity analysis). The
intent is that a hiring manager can browse the repository in 15 minutes
and immediately see depth across all SRE competencies.

---

## 2. Goals and Learning Objectives

### Primary Goals

- Practice the four SRE pillars hands-on with real failure modes, real
  alerts, and real postmortems — not simulated checkboxes.
- Build a public portfolio asset that demonstrates senior-level SRE
  thinking through artifacts (docs, runbooks, postmortems) rather than
  just configuration files.
- Deepen GitOps expertise (ArgoCD with ApplicationSets, sync waves,
  Argo Rollouts) beyond the typical "deploy an app" use case.
- Introduce and master tools the author has not used in production:
  Sloth, Chaos Mesh, Grafana OnCall, Kyverno, OpenCost, k6, Tempo.

### Secondary Goals

- Demonstrate **automation against toil** — the defining SRE practice —
  by building scheduled, autonomous jobs that perform recurring
  reliability work without human intervention.
- Tie multiple pillars together (e.g., chaos experiments triggered on a
  schedule, postmortem drafts auto-generated by long-running alerts) to
  show systems-level thinking, not just isolated tool knowledge.
- Produce written, narrative deliverables that demonstrate
  communication skill — a critical and often-missing element in
  SRE candidates.

### Non-Goals

- Cloud provider experience (AWS, GCP, Azure). Deferred to a future
  project. The author has decided to maximize SRE depth in this lab
  rather than dilute it with cloud breadth.
- Building a production-grade application. The demo app is treated as a
  given; all engineering effort focuses on operating it reliably.
- Multi-cluster federation. A single local cluster is sufficient to
  demonstrate every SRE concept in scope.

---

## 3. Constraints

- **Budget:** zero monetary cost. Local-only infrastructure.
- **Time:** approximately 5–8 hours per week, over ~18 weeks.
- **Hardware:** a developer laptop with at least 16 GB RAM and 6 CPU
  cores available to Docker.
- **Tooling philosophy:** open-source-first. Every component must be
  freely usable, self-hostable, and reproducible.
- **Showcase orientation:** every phase must produce a written
  artifact suitable for public consumption.

## 5. Architecture Overview

The system is conceptually divided into four layers:

**Cluster layer.** A single multi-node kind cluster simulates a small
production environment. Three worker nodes allow scheduling diversity,
realistic pod disruption budget testing, and node-level chaos
experiments.

**Platform layer.** Deployed via ArgoCD as a set of "platform"
applications — these provide the cross-cutting capabilities every
workload depends on: observability (metrics, logs, traces), GitOps
controllers, policy enforcement, on-call tooling, automation, cost
attribution.

**Application layer.** Google Online Boutique, plus one small custom
service authored as part of the project (to give the CI pipeline
something meaningful to build). Both are deployed via ArgoCD using the
App-of-Apps pattern.

**Automation layer.** Jenkins acts as the SRE automation hub, running
scheduled jobs that perform reliability work autonomously — disaster
recovery drills, chaos schedules, capacity baselining, postmortem
drafting. This is the unique differentiator of the project and is
described in detail in Section 9.

All inter-component wiring is declarative and lives in the repository.
A single bootstrap step brings up the cluster and ArgoCD; from that
point on, everything else converges automatically.

## 8. Phase Details

### Phase 0 — Bootstrap (Week 1)

Establish the minimum viable foundation. The goal is a single command
that brings up a working kind cluster with ArgoCD installed and the
Online Boutique reconciling from the repository. No observability, no
SLOs, nothing fancy — just a clean GitOps loop end to end.

Deliverables: project README first draft, initial architecture diagram,
bootstrap automation, App-of-Apps structure committed.

### Phase 1 — Observability Foundation (Weeks 2–3)

Deploy the full three-pillar observability stack via ArgoCD. Configure
Promtail to ship application logs to Loki. Configure the OpenTelemetry
Collector to receive traces from Online Boutique (which already emits
OTel) and forward them to Tempo. Build baseline Grafana dashboards
covering the RED and USE methods.

The phase ends when a developer can take a single failing request,
follow it from metrics → logs → trace in Grafana, and explain what
happened.

Deliverables: observability architecture documentation, baseline
dashboards, one walkthrough showing end-to-end correlation of metrics,
logs, and traces.

### Phase 2 — SLOs and Error Budgets (Weeks 4–5)

Define Service Level Indicators and Service Level Objectives for four
key services: frontend, checkout, cart, and product catalog. Use Sloth
to translate declarative SLO specs into Prometheus recording and alert
rules, including multi-window multi-burn-rate alerts as recommended by
the Google SRE workbook.

Write an explicit error budget policy: what happens when budget is
exhausted, who decides, how it is restored.

Deliverables: one SLO markdown per service, the error budget policy
document, Sloth specifications in the repository, working burn-rate
alerts in Alertmanager.

### Phase 3 — Incident Response (Weeks 6–7)

Deploy Grafana OnCall and wire Alertmanager to route burn-rate alerts
into it. Define a single-person on-call schedule (you), with realistic
escalation rules. Write a runbook for every Prometheus alert in the
system; each runbook must follow a consistent template (symptom,
diagnosis steps, mitigation, rollback, escalation).

At the end of the phase, run a deliberate game day: trigger a failure,
follow your own runbook, time the response, and write a real
postmortem. The postmortem template must include timeline, root cause,
contributing factors, what went well, what went poorly, and action
items.

Deliverables: Grafana OnCall configuration, eight to ten runbooks, a
postmortem template, one completed real postmortem.

### Phase 4 — SRE Automation Hub with Jenkins (Weeks 8–9)

See Section 9 for the full description.

### Phase 5 — Chaos Engineering (Weeks 10–11)

Install Chaos Mesh and design six experiments, each documented before
execution with a clear hypothesis. Experiment categories should cover:
pod-level failures (kill, evict), network conditions (latency, packet
loss, DNS chaos), resource pressure (CPU, memory), and node-level
disruption (taint and drain).

Crucially, integrate chaos with the Jenkins automation hub so that
experiments run on a regular schedule against the cluster, producing
ongoing evidence that the system tolerates the documented failure
modes.

Deliverables: one report per experiment in
`docs/chaos-experiments/` following a hypothesis-method-result-learning
structure. Scheduled chaos runs visible in Jenkins job history.

### Phase 6 — Progressive Delivery (Weeks 12–13)

Convert two services (cart and checkout) from standard Deployments to
Argo Rollouts. Define analysis templates that pull error-rate and
latency metrics from Prometheus during canary steps and automatically
roll back on regression.

The phase culminates in a deliberately bad deploy: a version that
returns elevated 500s on a fraction of requests. The rollout must
detect the regression mid-canary and roll back automatically, producing
a clean record in Argo Rollouts and an alert in Alertmanager.

Deliverables: Rollout manifests, AnalysisTemplate manifests, recorded
evidence of an automatic rollback, written narrative of the
bad-deploy exercise.

### Phase 7 — Policy as Code (Week 14)

Install Kyverno and author cluster policies that enforce reliability
requirements: every workload must declare liveness and readiness
probes, every Deployment must have a PodDisruptionBudget, every
container must declare resource requests and limits, images must come
from approved registries, and every workload must carry a label
linking it to its SLO.

The policies are intentionally tied back to reliability — they are not
generic security policies, but a codification of the lessons the
earlier phases will have taught.

Deliverables: Kyverno policies in the repository, a written
justification document linking each policy to a real reliability
concern.

### Phase 8 — Cost Visibility (Week 15)

Install OpenCost and build dashboards showing cost-per-namespace and
cost-per-service in the local cluster. Because the local cluster has no
real cost, write an extrapolation document estimating what the same
workload would cost on a small AWS EKS deployment. This demonstrates
FinOps awareness without needing a real cloud bill.

Deliverables: OpenCost dashboards, `cost-analysis.md` document with
extrapolated cloud costs and right-sizing recommendations.

### Phase 9 — Capacity Planning (Week 16)

Author k6 load scenarios at four levels: baseline (current production
load), 2x, 5x, and breakpoint (until the system fails). Run each
scenario, capture Prometheus data, and write a real capacity report
including breakpoint analysis, identified bottlenecks, HPA tuning
recommendations, and headroom estimates.

Deliverables: k6 scripts in the repository, `capacity-report.md`
containing graphs, breakpoints, and prioritized recommendations.

### Phase 10 — Polish and Writeups (Weeks 17–18)

Final pass on the README and top-level architecture diagram. Author
three blog-style writeups in `docs/blog/`: one on the SLO design
process, one on the most interesting chaos experiment, one on the
auto-rollback story. Review every document for clarity and consistency.

Deliverables: polished README, refined architecture diagram, three
long-form writeups suitable for sharing publicly.

## 9. Jenkins as the SRE Automation Hub

This section deserves dedicated treatment because the Jenkins
integration is the conceptual differentiator of the project.

### Framing

Most personal SRE projects use a CI tool to do exactly one thing: build
container images. That is a DevOps story, not an SRE story. In this
project, Jenkins plays the role of the **automation hub against toil**,
running the recurring reliability work that a real SRE team would
otherwise perform manually. The CI pipeline still exists, but it is the
smaller half of Jenkins' role.

Jenkins runs in-cluster, deployed via ArgoCD, using the Kubernetes
plugin to spawn ephemeral agent pods on demand. This is itself a
worthwhile pattern to demonstrate, distinct from the legacy
"long-lived agents on VMs" model.

### Role 1 — Continuous Integration

A single CI pipeline builds the one small custom service authored as
part of the project. The pipeline lints, runs unit tests, builds a
container image, scans it with Trivy, and pushes it to a local
registry. ArgoCD then picks up the new tag and reconciles. This piece
is intentionally minimal — its purpose is to show that the candidate
can build a modern Jenkins-on-Kubernetes pipeline, not to over-engineer
yet another CI demo.

### Role 2 — The SRE Automation Hub

Four scheduled jobs make Jenkins the engine of toil reduction in the
project. Each job writes its output as a timestamped markdown file
into `docs/automation-reports/`, so a reviewer can see actual evidence
that the automation runs and produces useful results.

**Job A — Nightly disaster recovery drill.** Once per night, the job
provisions a fresh sandbox namespace, restores Redis and the catalog
data from a known snapshot, runs a battery of smoke tests against the
restored state, and reports pass or fail. The report includes the time
to restore (RTO) and an indication of any data drift (RPO). Over time,
the repository accumulates a history of DR drills — visible proof that
recovery is exercised, not theoretical.

**Job B — Scheduled chaos experiments.** Once per week, the job picks
one of the documented chaos experiments and runs it against the
cluster, observes SLO compliance during and after, and reports whether
the system tolerated the fault as expected. This converts chaos from a
one-time demo into ongoing assurance.

**Job C — Weekly capacity baseline.** Once per week, the job runs a
baseline k6 scenario, captures key metrics from Prometheus, compares
the results against the previous week, and flags regressions. This
gives the project a real "performance over time" track record rather
than a single point-in-time capacity number.

**Job D — Auto-postmortem drafting.** When a Prometheus alert remains
firing for longer than a configurable threshold, a webhook triggers
Jenkins to pull the relevant Grafana panels, recent log windows, and
trace samples into a postmortem markdown template, pre-populated and
ready for a human to review and complete. This is the most ambitious
job in the set and demonstrates real automation thinking — reducing
the toil of postmortem drafting, not just incident response itself.

### Why this matters for interviews

The fourth automation job in particular is the kind of artifact that
sticks in an interviewer's memory. Almost no candidate produces an
end-to-end demonstration of toil reduction, with running automation
and a paper trail of results. This is the difference between
*claiming* to understand SRE and *doing* SRE.

## 10. Documentation Deliverables

Documentation is treated as a first-class deliverable, not an
afterthought. The following written artifacts will exist at the end of
the project:

- **README** — project pitch, architecture diagram, quick-start, and
  links to every major artifact.
- **Architecture document** — design decisions, component
  responsibilities, sequence diagrams for key flows.
- **Service Level Objective documents** — one per service, including
  SLI definition, SLO target, error budget policy, alert thresholds.
- **Runbooks** — one per alert, in a consistent template.
- **Postmortems** — real postmortems from game days, following a
  standard template.
- **Chaos experiment reports** — one per experiment, structured as
  hypothesis, method, observations, learnings.
- **Automation reports** — accumulated output of the four Jenkins
  scheduled jobs.
- **Capacity report** — k6 results, breakpoints, recommendations.
- **Cost analysis** — OpenCost data extrapolated to a cloud
  equivalent.
- **Policy justification** — written rationale for each Kyverno
  policy, linking back to specific reliability concerns.
- **Blog-style writeups** — three long-form posts narrating the most
  interesting parts of the project.

## 12. Risks and Mitigations

**Resource exhaustion on the local laptop.** Running kind plus the
full observability stack plus Jenkins plus the demo app will consume
significant RAM. Mitigations: start with the cheaper components first,
add the heavier ones only after the bootstrap is stable, drop Tempo if
necessary (traces are the most expensive component to retain), and
consider k3d as a lighter alternative to kind.

**Scope creep.** The project is ambitious. Phases 5 through 9 are
genuinely valuable but optional from an interview standpoint — phases 0
through 4 are the SRE core. If the schedule slips, drop later phases
rather than rushing earlier ones. A polished, complete Phase 4 is
worth more than a half-finished Phase 9.

**Synthetic postmortems.** It is tempting to invent dramatic incidents
to write good postmortems about. Interviewers can detect this. Run
real game days, let real things break, and write about what actually
happened — even when it is mundane.

**Documentation drift.** As the project evolves, early documents will
become stale. Build in a quick documentation review at the end of every
phase to catch drift before it accumulates.

**Showcase paralysis.** Aiming for "perfect" can prevent shipping
anything. Push the repository public after Phase 1 and update it
continuously. Visible progress is more impressive than a long-delayed
big bang.

## 13. Success Criteria

The project is successful when all of the following are true:

- The repository is public on GitHub.
- A single bootstrap command brings the entire stack up on a fresh
  laptop.
- All four core SRE pillars (SLO, observability, chaos, incident
  response) are represented by both working code and written
  artifacts.
- The four Jenkins automation jobs are operational and have produced
  at least four weeks of real reports.
- At least one real postmortem from a real game day exists in the
  repository.
- The README enables a hiring manager unfamiliar with the project to
  understand its purpose, browse its artifacts, and form a positive
  impression in under fifteen minutes.
